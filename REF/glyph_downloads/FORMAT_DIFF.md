# Glyph 59 与当前格式差异说明

## 对比对象

- `REF/glyph_downloads/59/glyph.csv`
- `REF/glyph_downloads/59/glyph_line.csv`
- `app/src/main/java/moe/lyniko/glyphhacker/glyph/GlyphDictionary.kt` 中 `RAW` 常量（当前程序使用）

## 59 版本数据格式

### 1) `glyph.csv`

- 无表头，逗号分隔。
- 每行最多 5 个词位（不足补空）。
- 示例：`gain,truth,open,human,soul`

### 2) `glyph_line.csv`

- 无表头，`词名,编码`。
- 编码是**路径节点序列**（不是边对列表）。
  - 例如：`all,018a920` 表示路径 `0 -> 1 -> 8 -> a -> 9 -> 2 -> 0`
  - 实际边集合由相邻点推导：`01,18,8a,a9,92,20`
- 节点字符范围为 `0-9` + `a`（共 11 个点）。

## 当前程序数据格式（`GlyphDictionary.RAW`）

- 每行格式：`编码|别名1,别名2,...`
- 编码是**边对列表**：每 2 个字符表示一条边。
  - 例如：`010512233445|All`
- 代码中使用 `digitToInt(11)` 解析，节点同样为 `0-9` + `a`。

## 核心差异

### 1) 编码语义不同

- `59/glyph_line.csv`：路径序列（stroke/path）。
- 当前 `RAW`：边集合（edge set）。

### 2) 分隔符与字段不同

- 59：CSV（`,`）。
- 当前：`|` 分隔编码与别名，别名内再用 `,` 分隔。

### 3) 点位编号存在重映射（59 -> 当前）

在把 59 的路径转为边集合后，对 133 个同名词做匹配，可得到一组稳定编号映射（132/133 完全一致）：

`0->0, 1->5, 2->1, 3->9, 4->6, 5->a, 6->8, 7->7, 8->4, 9->2, a->3`

这说明两份数据不仅编码语义不同，点位编号体系也不是同一套直映射。

### 4) 词名规范不同

- 59 更偏向机器 key（如 `open_all`, `clear_all`, `live_again`, `nzeer`）。
- 当前格式有大量同义词/自然语言别名（如 `Open All`, `Close All`, `Live Again`, `N'zeer`）。

## 兼容转换建议

若要把 `59/glyph_line.csv` 转为当前可直接使用的编码，步骤应为：

1. 将路径序列转为相邻边集合。
2. 应用点位映射（59 -> 当前）。
3. 对边做无向归一化（小编号在前）并去重。
4. 最后按当前格式写为 `编码|别名...`。

备注：在上述映射下，公共词中仅 `n` 出现 1 处边差异，疑似数据版本/词条定义差异。
